Architecture Overview

DimeTech operates as an independent verification and middleware API layer that sits between the POS system and the payment processor. It intercepts the transaction total, appends 11¢, routes funds to verified certified charities (10¢) and operations (1¢), generates audit-ready tax receipts, and returns a modified receipt — all without touching the merchant’s existing workflow. As an independent third party, DimeTech provides the verification trail that enterprises need when claiming charitable tax deductions.

Key insight: DimeTech doesn’t replace any POS system — it sits on top as an independent verification layer. The adapter handles POS differences; the core handles fund routing, audit trail, and tax compliance. Enterprises need a third party for this: when the CRA (Canada Revenue Agency) audits a $13M+ charitable deduction, they point to DimeTech’s verified records.

The Adapter Pattern

Every POS system has a different API, webhook format, and data model. DimeTech solves this with a per-POS adapter that normalizes everything into a standard internal format before it reaches the core logic.

POS Variation What Differs What DimeTech Normalizes
Webhook format Toast uses REST callbacks, Square uses event subscriptions Standard event schema regardless of source
Auth model OAuth 2.0, API keys, session tokens Unified credential store per merchant
Transaction data Field names, nesting, currency handling Canonical transaction object
Receipt format HTML, JSON, thermal printer commands DimeTech line item injected in native format
Permissions Varies by marketplace/API tier Minimum required scopes per adapter

API Specification

The DimeTech API exposes a simple REST interface. POS adapters call this internally, but it can also be called directly for custom integrations.

POST /v1/transactions

Called when a transaction is completed. Appends the DimeTech contribution (not a surcharge — built into the price) and returns the split.

// Request POST /v1/transactions Content-Type: application/json { "merchant_id": "mch_uber_eats_001", "pos_system": "toast", "transaction_id": "txn_8847291", "amount_cents": 1873, "currency": "CAD", "timestamp": "2026-04-15T14:32:00Z" }
// Response { "status": "success", "dimetech_id": "dt_0847291", "original_amount_cents": 1873, "contribution_cents": 11, // not a surcharge — built into the price "final_amount_cents": 1884, "split": { "charity_cents": 10, "operations_cents": 1 }, "charity_recipient": "Daily Bread (Certified Charity)", "tax_receipt_url": "https://api.dimetech.io/receipts/dt_0847291", "audit_hash": "sha256:9f3c...a1e7" }

GET /v1/merchants/:id/summary

Returns aggregate donation totals and tax deduction estimates for a merchant or partner.

// Response { "merchant_id": "mch_uber_eats_001", "period": "2026-Q1", "total_transactions": 142300, "total_donated_cents": 1423000, "estimated_tax_deduction_cents": 284600, "receipts_issued": 142300, "audit_status": "verified" }

Why the Middleware Approach Wins

Approach Pros Cons
Replace POS Full control Merchants won’t switch. Dead on arrival.
POS Plugin Native integration Must build per-POS app. Marketplace approval. Slow.
Middleware API Universal. Fast to deploy. POS-agnostic core. Requires adapter per POS. Webhook reliability.

DimeTech uses the middleware approach because it’s the only one that scales to “all POS systems” without requiring merchant buy-in on a system change. The adapter layer absorbs the complexity; the core stays clean.
